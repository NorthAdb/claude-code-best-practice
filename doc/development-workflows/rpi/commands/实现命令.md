# 实现命令（中文翻译）
> 中文翻译自 [development-workflows/rpi/.claude/commands/rpi/implement.md](https://github.com/NorthAdb/claude-code-best-practice/blob/main/development-workflows/rpi/.claude/commands/rpi/implement.md)

---
description: Execute phased implementation with validation gates
argument-hint: "<feature-slug> [--phase N] [--validate-only]"
---

## 用户输入

```text
$ARGUMENTS
```

你**必须**解析用户输入以提取功能slug（`rpi/`中的文件夹名称）。

## 目的

此命令执行基于规划文档的分阶段功能实现。它协调专门的Agent，强制执行验证关卡，并确保在整个实施过程中符合宪法。

**前置条件**：
- 功能文件夹存在于 `rpi/{feature-slug}/`
- 规划已完成（`rpi/{feature-slug}/plan/PLAN.md` 存在）

**输出位置**：`rpi/{feature-slug}/implement/`

**这是RPI工作流的第4步**（最后一步——实际实现）。

## 标志

- `--phase N`：执行特定的阶段编号（1-8），如果省略则从阶段1开始
- `--validate-only`：仅验证当前阶段，不实施
- `--skip-validation`：跳过验证关卡直接进行（谨慎使用）

## 可用Agent

所有Agent使用**Opus模型**以获得最高质量。

### 实施Agent

| Agent | 类型 | 使用时机 |
|-------|------|----------|
| `senior-software-engineer` | 自定义 | 所有实施任务 |

### 支持Agent

| Agent | 类型 | 目的 |
|-------|------|------|
| `Explore` | 内置 | 实施前的代码探索 |
| `code-reviewer` | 自定义 | 代码审查和质量验证 |
| `constitutional-validator` | 自定义 | 验证是否符合项目宪法 |
| `documentation-analyst-writer` | 内置 | 文档生成 |

### Agent路由

所有实施任务由 `senior-software-engineer` Agent处理。

## 阶段0：加载上下文和规则

**前置条件**：从用户输入解析出功能slug

**流程**：

### 0.1 加载项目宪法

1. 在仓库中查找宪法或原则文档
2. 如果存在，提取：
   - 技术约束（类型安全、测试、组件隔离）
   - 业务原则（质量标准、工作流）
   - 架构边界
3. 存储约束以便在实施过程中强制执行

### 0.2 加载领域特定指南

根据要修改的文件，加载相关的项目指南：
- 检查组件特定的README文件
- 检查编码风格指南
- 检查测试需求文档

### 0.3 分析实施范围

1. 读取 `rpi/{feature-slug}/plan/PLAN.md`
2. 识别所有要修改的文件
3. 将文件映射到实施Agent

**输出**：
- 宪法上下文摘要
- 已加载的领域规则
- 文件到Agent的映射
- 阶段执行计划

**验证**：
- [ ] 宪法已加载（如存在）
- [ ] 受影响文件的领域规则已加载
- [ ] 所有文件已映射到Agent
- [ ] 执行计划已理解

## 分阶段实施工作流

### 阶段实施循环

对于PLAN.md中的每个阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│ 阶段 N：[阶段名称]                                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 代码发现 (Explore Agent)                                     │
│     └─→ 在修改前理解现有代码                                       │
│                                                                  │
│  2. 实施 (senior-software-engineer)                              │
│     └─→ 实施阶段交付物                                            │
│                                                                  │
│  3. 自我验证                                                     │
│     └─→ 工程师根据阶段清单进行验证                                  │
│                                                                  │
│  4. 代码审查 (code-reviewer Agent)                               │
│     └─→ 安全性、正确性、可维护性                                    │
│                                                                  │
│  5. 用户验证关卡                                                  │
│     └─→ 停止并请求用户批准                                        │
│         ├─→ 通过：继续下一个阶段                                   │
│         ├─→ 有条件通过：记录问题，继续                              │
│         └─→ 失败：修复问题，重新验证                                │
│                                                                  │
│  6. 文档更新                                                     │
│     └─→ 更新PLAN.md中的阶段状态                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 步骤1：代码发现（每个阶段）

**Agent**：Explore（内置，通过Task工具）

**目的**：在进行更改之前，让实施扎根于代码现实。

**流程**：
1. 通过Task工具使用 `subagent_type="Explore"` 启动Explore Agent
2. 请求分析受当前阶段影响的文件
3. 理解现有模式、集成点、约束

**Explore Agent提示**：
```
分析代码库，为实施[功能名称]的阶段N做准备。

此阶段要修改的文件：
[从PLAN.md列出文件]

调查并记录：

1. **当前实现**
   - 这些文件目前如何工作？
   - 使用了哪些模式？
   - 哪些函数/类将受到影响？

2. **集成点**
   - 哪些其他文件导入或使用这些模块？
   - 哪些API或接口将发生变化？
   - 哪些测试覆盖了此代码？

3. **依赖项**
   - 使用了哪些库？
   - 有哪些内部工具可用？
   - 当前代码存在哪些约束？

4. **要遵循的模式**
   - 这些文件使用了哪种编码风格？
   - 遵循了哪些命名约定？
   - 存在哪些错误处理模式？

5. **风险和注意事项**
   - 更改此内容可能导致什么问题？
   - 存在哪些边界情况？
   - 有哪些向后兼容性问题？

提供发现摘要以指导实施。
```

**输出**：供实施Agent使用的发现摘要

## 步骤2：实施（每个阶段）

**Agent**：senior-software-engineer

**流程**：
1. 使用senior-software-engineer Agent
2. 提供来自步骤1的发现上下文
3. 实施该阶段的所有交付物
4. 遵循宪法约束和项目规则

**实施Agent提示模板**：
```
以[agent名称]Agent的身份，为[功能名称]实施阶段N的交付物。

## 上下文
- 宪法约束：[来自阶段0]
- 领域规则：[来自阶段0]
- 发现摘要：[来自步骤1]

## 阶段N交付物
[从PLAN.md列出]

## 要修改的文件
[从PLAN.md列出文件及具体变更]

## 实施要求
1. 遵循发现过程中识别的现有代码模式
2. 遵守宪法约束（类型安全、测试等）
3. 遵循项目特定规则（如适用）
4. 为新功能编写测试
5. 包含适当的日志记录
6. 优雅地处理错误

## 质量检查清单
- [ ] 代码遵循现有模式
- [ ] 类型注解在适用处存在
- [ ] 测试已编写并通过
- [ ] 现有功能无破坏性变更
- [ ] 已添加日志记录以支持可观测性
- [ ] 错误处理全面

实施所有交付物并报告已完成的工作。
```

## 步骤3：自我验证

**Agent**：senior-software-engineer（与步骤2相同）

**流程**：
1. Agent根据阶段检查清单验证实施
2. 运行代码检查（使用项目配置的检查工具）
3. 运行与更改相关的测试
4. 验证构建成功

**验证命令**（根据你的项目进行调整）：

```bash
# 运行检查工具
[your-linter-command]

# 运行测试
[your-test-command]

# 构建/编译
[your-build-command]
```

**自我验证检查清单**：
- [ ] 所有交付物已实施
- [ ] 代码检查通过
- [ ] 测试通过
- [ ] 构建成功
- [ ] 现有测试中无回归
- [ ] 宪法约束得到遵守
- [ ] 领域规则得到遵循

## 步骤4：代码审查

**Agent**：code-reviewer（自定义，自动调用）

**流程**：
1. 调用code-reviewer Agent审查更改
2. 重点关注正确性、安全性、可维护性
3. 在继续之前解决阻塞性问题

**代码审查Agent提示**：
```
以code-reviewer Agent的身份，审查[功能名称]的阶段N实施。

## 更改的文件
[列出修改的文件]

## 所做的更改
[实施摘要]

## 审查重点
- 正确性和测试
- 安全性和依赖卫生
- 架构边界
- 清晰性优于花哨

## 宪法约束
[来自阶段0]

使用标准输出格式提供审查。
```

**审查裁决**：
- **批准**：进行用户验证
- **附带建议批准**：记录建议，继续进行
- **需要修改**：修复问题，重新审查

## 步骤5：用户验证关卡

**关键**：此步骤**需要**用户交互。**不要**自动进行。

**流程**：
1. 展示阶段交付物检查清单
2. 显示已实施的内容（更改的文件、添加的功能）
3. 展示PLAN.md中的验证标准
4. 展示代码审查结果
5. **停止并等待用户决策**

**验证请求格式**：
```
## 阶段N验证请求

### 已完成的交付物
- [x] [交付物1] - [实施摘要]
- [x] [交付物2] - [实施摘要]
- ...

### 更改的文件
| 文件 | 更改类型 | 行数 |
|------|----------|------|
| [file] | [add/modify] | [±N] |

### 测试
- [x] 单元测试：通过
- [x] 集成测试：通过
- [x] 构建：成功

### 代码审查
- 裁决：[批准 / 附带建议批准]
- 问题：[无 / 列表]

### 验证标准（来自PLAN.md）
- [ ] [标准1]
- [ ] [标准2]
- ...

---

**请验证阶段N：**
- **通过**：阶段完成，继续阶段N+1
- **有条件通过**：在下方注明问题，谨慎继续
- **失败**：在继续前指定要修复的问题
```

**用户决策**：
- **通过**：继续下一个阶段
- **有条件通过**：记录问题，继续下一个阶段
- **失败**：修复问题，重新运行步骤2-5

---

## 步骤6：文档更新

**流程**：
1. 用阶段状态更新 `rpi/{feature-slug}/plan/PLAN.md`
2. 用验证结果更新 `rpi/{feature-slug}/implement/IMPLEMENT.md`
3. 将每个阶段的验证附加到IMPLEMENT.md

### 阶段状态跟踪

更新PLAN.md中的复选框：
```markdown
- [ ] 阶段N：未开始
- [~] 阶段N：进行中
- [x] 阶段N：已验证（通过）
- [!] 阶段N：有条件通过（附备注）
- [-] 阶段N：验证失败（需要返工）
```

### IMPLEMENT.md模板

```markdown

**功能**：[feature-slug]

**状态**：[进行中 / 已完成]

[列出并附行数]

### 测试结果
```

## 错误处理

### 实施失败

**如果实施失败**：
1. 记录具体的失败
2. 分析根本原因
3. 尝试替代方法（最多2次尝试）
4. 如果仍然失败，**停止**并向用户寻求指导
5. 不要在实施损坏的情况下继续下一个阶段

**消息**："实施失败：[错误]。尝试了[N]种方法。需要用户指导。"

### 测试失败

**如果测试失败**：
1. 分析失败原因（代码错误 vs 测试错误）
2. 修复问题
3. 重新运行测试
4. 如果持续失败，记录并向用户求助
5. 不要在测试失败的情况下标记阶段完成

**消息**："测试失败：[失败详情]。已尝试修复但未成功。需要用户审查。"

### 构建失败

**如果构建失败**：
1. 检查类型错误
2. 检查缺失的导入
3. 检查语法错误
4. 修复并重新构建
5. 如果持续失败，升级给用户处理

**消息**："构建失败：[错误]。无法自动解决。"

### Agent失败

**如果Agent失败或超时**：
1. 使用相同的输入重试一次
2. 如果仍然失败，在不使用该Agent贡献的情况下继续
3. 在验证请求中记录差距

**消息**："Agent [名称] 失败。在不使用其贡献的情况下继续。"

## 备注

### 何时使用此命令

- 在 `/rpi:plan` 生成PLAN.md之后
- 当需要带有验证关卡的分阶段实施时
- 对于需要结构化实施的功能

### 何时**不**使用此命令

- 错误修复（太重了，直接修复即可）
- 非常简单的更改（少于30分钟的工作）
- 探索性原型开发
- 仅文档更改

### 最佳实践

1. **先审查PLAN.md**：了解你要实施的内容
2. **信任代码发现**：让Explore Agent指导实施
3. **遵循现有模式**：让代码发现指导实施
4. **不要跳过验证**：关卡存在是为了及早发现问题
5. **边做边记录**：每个阶段后更新状态
6. **卡住时提问**：宁可提问也不要错误地继续

### RPI工作流的一部分

步骤4 / 4（描述 → 研究 → 规划 → **实施**）

## 完成后的操作

**重要**：完成实施后（所有阶段或取得重大进展后），始终提示用户压缩对话：

> **上下文管理**：此实施工作流消耗了大量上下文。为了保留进度并释放空间，请运行：
>
> ```
> /compact
> ```
>
> 这将总结对话并保留实施状态，同时减少未来工作的令牌消耗。

**何时提示压缩**：
- 所有阶段完成后
- 完成每个主要阶段后（如果是多会话实施）
- 如果在实施过程中上下文不足
