# 为什么 Harness 如此重要

> 为什么 Claude Code 的功能不是"披着外衣的提示词"——以及为什么 Harness 才是区分玩具输出和生产级工程工作的关键。

---

## 核心结论

Claude Code 老用户中流传着一个常见的**简化论**：*"skills、commands、subagents、hooks——它们最终都会变成模型的提示词，所以只靠一个强有力的提示词就够了。"*

在模型最终推理调用的层面上，这在技术上是正确的。模型只看到 token。

但在所有其他层面——真正的软件工程发生的地方——**这个简化论站不住脚。** Harness 不是一个提示词投递系统。它是一个**提示词构建系统、一个确定性执行系统、一个上下文架构系统**——而这些能力是无法通过更强硬的措辞来替代的。

本报告解释了这个简化论正确的地方、失败的地方，以及为什么混淆"模型看到什么"和"系统做什么"会让从业者远离那些真正赋予 Claude Code 杠杆作用的功能。

---

## 听起来很对的简化论

对于一个**单次原子任务**——"给我写一个递归斐波那契函数"——harness 对输出质量没有任何贡献。把同样的 token 交给同一个模型，无论它们是通过 skill、command 还是原始提示词到达的，你都会得到相同的输出分布。

在这个狭窄的范围内，简化论成立：

> 输出质量 ≈ 提示词质量

这是 Claude Code 相比普通聊天机器人几乎没有价值的范围。这恰恰也是简化论隐含假设的范围——而且正是**真正的工程工作不在的地方**。

---

## 简化论在哪些地方失效

Harness 的十个架构能力在提示词无法触及的层面运作：

| # | 能力 | 作用 | 为什么提示词无法复制 |
|---|------|------|---------------------|
| 1 | **上下文隔离** | 子代理在独立的上下文窗口中运行 | 一个提示词填满一个窗口。N 个并行子代理提供约 N× 的有效上下文 |
| 2 | **Harness 强制的工具限制** | `allowed-tools` / `disallowedTools` 在模型使用工具之前就阻止了它们 | 提示词中的指令是建议性的；模型可以忽略。Deny 规则无法被忽略 |
| 3 | **延迟加载的规则和记忆** | `paths:` frontmatter 和后代 `CLAUDE.md` 只在 Claude 触及匹配路径时加载 | 提示词是静态的——它无法根据运行时正在读取的文件而有条件地加载 |
| 4 | **钩子：确定性代码执行** | Shell 命令在生命周期事件（PreToolUse、PostToolUse、Stop 等）触发，并且可以**阻止**工具调用 | 提示词无法拦截自己的工具调用。即使模型不"愿意"，钩子也会执行 |
| 5 | **模型路由** | `model: haiku` 或 `model: opus` 将调用路由到不同的模型端点 | 提示词中的任何 token 都无法改变回答问题的模型 |
| 6 | **并行性** | 多个子代理并发执行 | 提示词是顺序的。Harness 调度并收集并行进程的结果 |
| 7 | **跨会话持久化** | 记忆系统和设置层级结构跨对话持久化 | 提示词在会话结束时消亡 |
| 8 | **模块化系统提示** | CLI 根据激活的功能有条件地加载 110+ 个系统提示片段 | 用户无法手动编写或替换 CLI 内部的提示片段 |
| 9 | **Skill 预加载** | `skills:` 字段将 skill 的完整内容注入子代理的启动上下文 | 用户无法预填充另一个 agent 的上下文——只有 harness 加载器可以 |
| 10 | **权限分类** | `auto` 权限模式使用后台分类器预批准或阻止工具调用 | 提示词无法为自己添加执行前安全层 |

每一行都是一个"强力措辞"绝对无法替代的维度。

---

## "提示词"这个词的两种用法

这个简化论利用了语义含混。"提示词"这个词被用来指代两个截然不同的东西：

| 含义 | 谁控制它 | 大小 |
|------|---------|------|
| (a) 用户输入的文字 | 用户 | ~6–60 token |
| (b) 模型在推理时看到的内容 | Harness | ~5,000–50,000+ token |

在聊天机器人中，(a) 和 (b) 是同一个东西。

在 Claude Code 中，它们有着根本的不同。

Harness 的职责恰恰是让 (b) 比 (a) 丰富得多：

```
用户输入: "write a recursive flatten function"   ← (a) ~6 tokens

模型在推理时实际看到的内容:                     ← (b) ~15,000 tokens
  ├── CLAUDE.md（项目约定）
  ├── 匹配的 .claude/rules/*.md（通过 paths: frontmatter 加载）
  ├── 模块化系统提示片段
  ├── 工具定义
  ├── 环境上下文（cwd、git 状态、平台）
  ├── 之前的对话历史
  ├── 模型通过 Read/Grep 工具读取的文件
  └── 用户的 6 个 token 请求
```

**输出质量是 (b) 的函数，而不是 (a) 的函数。** Harness 构建了 (b)。仅靠"强大的提示词"无法重现 (b)，因为其中的大部分内容不是用户编写的。

---

## 即使就输出质量而言，Harness 也在做功

考虑同一个提示词——"write a recursive flatten function"——在三种环境中：

| 环境 | 模型看到的 | 典型结果 |
|------|-----------|---------|
| 聊天机器人，无工具 | 这句话 | 教科书式的递归，通用风格 |
| Claude Code，不读取文件 | 句子 + CLAUDE.md | 匹配声明的项目约定 |
| Claude Code，智能循环 | 句子 + CLAUDE.md + 读取相邻文件 + 运行测试 | 匹配实际代码库的模式，通过测试，处理了现有代码处理的边界情况 |

同一个模型。同一个用户提示词。**三种不同的输出质量。** 差异在于 harness——具体来说，是它组装的有效上下文和它启用的迭代循环。

对于非平凡的任务，输出质量是以下因素的函数：

```
输出质量 = f(有效上下文, 模型能力, 迭代循环)
```

用户控制着 *有效上下文* 的一小部分（他们输入的提示词）。Harness 控制着其余部分——以及整个迭代循环。

---

## 简化论正确和错误的地方

| 主张 | 判断 |
|------|------|
| "在推理时，模型只看到 token。" | ✅ 正确 |
| "Skills、commands 和 subagent 提示词都为某个上下文贡献了 token。" | ✅ 正确 |
| "对于真空中的原子任务，提示词质量主导输出质量。" | ✅ 正确 |
| "因此，一个强大的提示词等同于使用各种功能。" | ❌ 错误 |
| "因此，harness 对输出质量不重要。" | ❌ 对于真正的工程任务是错误的 |

前三句话是准确的观察。跳跃到第四句才是推理失败的地方：它混淆了模型与包裹模型的系统，并混淆了原子任务与真正的工程工作。

---

## 正确的心智模型

> **提示词控制模型被要求做什么。**
> **Harness 控制在模型无法触及的层面系统做什么**——在 token 到达之前、token 产生之后、跨会话、跨上下文和跨进程。

功能不是多了几步的提示词。它们是 **harness 级别的原语**——确定性执行、上下文架构和基础设施路由——在模型无法置喙的层面运作。

一个有用的类比：

| 层面 | 聊天机器人 | Claude Code |
|------|-----------|-------------|
| 配方 | 用户的消息 | 用户的消息 + harness 组装的上下文 |
| 厨房 | 没有——只有一个学生 | 工具、钩子、记忆、并行工作者、生命周期事件 |

你可以写出世界上最好的配方。但没有厨房，你无法大规模烹饪。

---

## 给实践者的启示

1. **对于原子问题，提示词质量就是一切。** Harness 无关紧要。如果这就是你需要的所有，用聊天机器人就行。
2. **对于真正的代码库工作，Harness 在做无声的重活。** 推理时的有效提示词主要是 harness 构建的，而不是用户输入的。
3. **用功能来做提示词绝对做不到的事情：** 确定性（钩子）、隔离（子代理）、延迟加载（带 `paths:` 的规则）、持久化（记忆）、路由（每 agent 的 `model:`）和并行性。
4. **强大的提示词是必要的，但不是充分的。** 功能提供了提示词无法提供的确定性、隔离性和组合性。两者是互补的，而不是替代关系。
