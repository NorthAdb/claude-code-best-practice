# 🔥 Claude Code Ops 4.6 分析——高推理

当 Anthropic 发布像 Opus 4.6 这样的模型时，**模型权重**——数十亿个学习参数——是被冻结的。训练成本极其高昂（数百万美元，数周计算时间）。没有人会在夜间重新训练模型。

但权重只是一个更大系统中的一层。研究揭示了至少 **7 种不同的机制**，即使在模型权重被冻结的情况下，也可能导致真实或感知到的质量变化。

| 问题 | 答案 |
|----------|--------|
| 模型权重在发布后会改变吗？ | **不会**——所有提供商均已确认 |
| 模型日常表现会不同吗？ | **会**——已证实存在 ±8-14% 的差异 |
| 是有意的"削弱"吗？ | **不是**——没有证据表明存在故意的退化 |
| 基础设施 Bug 真实存在吗？ | **存在**——Anthropic 确认了 3 个 Bug，影响高达 16% 的请求 |
| 部分原因是心理因素吗？ | **是的**——确认偏差和蜜月效应是真实存在的 |
| 系统提示/训练后调整会改变吗？ | **会**——各提供商均有记录 |
| 用户应该相信自己的感知吗？ | **部分相信**——确实存在真实原因，但感知会放大它们 |

## 完整的推理栈

模型权重是冻结的，但其上的 **九层** 都可能独立影响你的体验：

```
┌──────────────────────────────────────────────┐
│  你的会话上下文                              │  ← 在会话内退化
│  （累积的错误、长对话）                      │
├──────────────────────────────────────────────┤
│  系统提示                                    │  ← 定期更新
│  （安全规则、行为指令）                      │
├──────────────────────────────────────────────┤
│  训练后（RLHF / 微调）                      │  ← 可以静默更新
│  （指令遵循、安全对齐）                      │
├──────────────────────────────────────────────┤
│  采样参数                                    │  ← 可在服务端调整
│  （temperature、top-p、top-k）                │
├──────────────────────────────────────────────┤
│  推测解码                                    │  ← 草稿模型质量变化
│  （草稿模型预测 + 验证）                     │
├──────────────────────────────────────────────┤
│  MoE 路由 / 批处理组成                       │  ← 已验证 ±8-14% 差异
│  （每个请求激活哪些专家）                    │
├──────────────────────────────────────────────┤
│  硬件路由                                    │  ← TPU vs GPU vs Trainium
│  （服务你的请求的是哪个集群）               │
├──────────────────────────────────────────────┤
│  量化级别                                    │  ← 可能在负载下变化
│  （FP16 vs INT8 vs INT4 精度）               │
├──────────────────────────────────────────────┤
│  编译器和运行时                              │  ← XLA Bug 已被证实
│  （XLA:TPU、CUDA、硬件特定代码）             │
├──────────────────────────────────────────────┤
│  模型权重（已冻结）                          │  ← 这些不会改变
│  （数十亿学习参数）                          │
└──────────────────────────────────────────────┘
```

关键的思维模型：**冻结的权重 ≠ 固定的行为**。这就像说"同一台发动机 = 同样的驾驶体验"却忽略了轮胎、路况、燃油质量和驾驶员疲劳。

## 已证实的成因：基础设施 Bug

### Anthropic 2025 年 9 月的事后分析

2025 年 9 月，Anthropic 发布了一份详细的事后分析，揭示了 **三个独立的基础设施 Bug**，这些 Bug 在 2025 年 8 月至 9 月期间降低了 Claude 的质量。他们的官方声明：

> "我们从不会因为需求、时间段或服务器负载而降低模型质量。用户报告的问题完全由基础设施 Bug 导致。"

### Bug #1 — 上下文窗口路由错误

Sonnet 4 的请求被错误地路由到配置了 1M Token 上下文窗口的服务器，而非标准服务器。

- **时间线**：8 月 5 日引入，8 月 29 日负载均衡变更后恶化
- **峰值影响**：最严重时（8 月 31 日）16% 的 Sonnet 4 请求受影响
- **用户影响**：约 30% 的 Claude Code 用户至少收到一条退化消息
- **隐蔽细节**：路由是"粘性的"——一旦你命中一台错误服务器，后续请求会持续发送到那里
- **修复**：9 月 4 日至 18 日（在各平台逐步推出）

### Bug #2 — TPU 输出损坏

TPU 服务器上的配置错误导致 Token 生成过程中出错，将高概率分配给本应很少出现的 Token。

- **症状**：英文回复中夹杂泰语或中文字符，明显的代码语法错误
- **受影响**：Opus 4.1 和 Opus 4（8 月 25-28 日）、Sonnet 4（8 月 25 日至 9 月 2 日）
- **范围**：仅 Claude API；第三方平台不受影响
- **修复**：9 月 2 日回滚

### Bug #3 — XLA:TPU 编译器误编译（最棘手的）

修复精度问题的代码变更意外暴露了 Google XLA:TPU 中的一个**潜在编译器 Bug**。

- **根本原因**：近似 top-k 操作（用于选择最可能的后续 Token）"有时返回完全错误的结果，但仅针对某些批大小和模型配置"
- **难以发现的原因**：其行为取决于在其之前或之后运行的操作，以及是否启用了调试工具
- **隐藏数月**：2024 年 12 月的一个先前临时方案意外地掩盖了这个更深层的 Bug
- **受影响**：已确认 Haiku 3.5；怀疑部分 Sonnet 4 和 Opus 3
- **解决方案**：从近似 top-k 切换到精确 top-k；接受"轻微效率影响"，因为"模型质量是不可妥协的"

### 为何检测如此困难

Anthropic 自己的自动评估未能捕捉到用户报告的质量退化，"部分原因是 Claude 通常能很好地从孤立错误中恢复"。每个 Bug 在不同平台上以不同速率产生不同症状，形成了"令人困惑的报告混杂，无法指向单一原因。"

关键背景：Claude 运行在**三种不同的硬件平台**上（AWS Trainium、NVIDIA GPU、Google TPU），每种硬件都有不同的故障模式、编译器和精度行为。你的请求在不同天数可能命中不同的硬件。

## 已证实的成因：MoE 路由差异

现代大型模型通常使用**混合专家（MoE）**架构，其中只有一部分模型参数（"专家"）为每个输入激活。一个学习后的路由器决定使用哪些专家。

Scale AI 的研究揭示了一个关键发现：

> "稀疏 MoE 与批推理的结合会创建不可预测的结果，因为批次的组成决定了你的查询被路由到哪个专家，而同一批次中其他用户的查询组合不是确定性的。"

### 各提供商的日常差异测量

| 提供商 | 日常得分差异 |
|----------|--------------------------|
| OpenAI（GPT-4 系列） | ±10-12% |
| Anthropic（Claude 系列） | ±8-11% |
| Google（Gemini 系列） | ±9-14% |

具体示例：同一模型某天的越狱抵抗力得分为 **77%，第二天为 63%**。同一模型、相同权重、相同测试——仅因基础设施就产生了 14 个百分点的波动。

这意味着即使零 Bug、零变更，同一模型在不同天数也可能仅因请求如何批处理和路由而产生显著不同的质量输出。当日常噪声为 10-15% 时，A/B 测试无法可靠检测出 5% 的质量信号。

## 已证实的成因：系统提示与训练后更新

### 系统提示变化

模型权重不会改变，但包裹这些权重的**系统提示**可以随时更新。对 Claude 系统提示演进的分析显示有数十次迭代，其中"热修复"——为修补不良行为而添加的简短指令——被定期添加和删除。

Claude 3.7 的系统提示包含多条针对常见 LLM"陷阱"的热修复指令。Claude 4.0 的系统提示移除了所有热修复指令，相关行为在训练后通过强化学习解决。

### 训练后调整理论

对无法解释的质量变化最合理的理论是：公司可以在不改变基础模型权重的情况下更新**微调和 RLHF**（基于人类反馈的强化学习）。这在技术上使得声称"模型没有变化"成为事实，同时通过更新的安全护栏和指令遵循调整改变了行为。

## 已证实的成因：静默模型切换

OpenAI 有多次被记录的静默更改用户交互模型的行为：

- 过夜移除模型选择器，迫使用户从 GPT-4o 切换到 GPT-5
- 让 GPT-4o 成为隐藏的"旧版模型"，需要在设置中手动切换，且应用内无任何通知
- 一个"自动切换器"Bug 将用户路由到错误的模型
- Plus 订阅用户报告模型在未经同意的情况下切换到"受限版本"

Sam Altman 承认部署过程"比我们希望的更波折一些"。Reddit 帖子获得了数千个赞，称新模型是一场"灾难"和"降级"。

这表明模型切换在行业中**确实会发生**——有时是有意的（产品决策），有时是意外的（路由 Bug）。

## 促成因素

### 负载下的量化

为了以经济高效的方式服务数百万用户，提供商可能会提供**量化**版本的模型——将精度从 FP16 降低到 INT8 或 INT4。这可以将内存使用减少 2-4 倍并加速推理，但会引入微小的质量损失。提供商是否会在负载下动态切换量化级别尚有争议，但技术上这种能力是存在的，并且在 vLLM 和 TensorRT 等服务框架中有充分记录。

### 推测解码

现代服务栈使用较小的"草稿"模型预测多个后续 Token，然后由真实模型验证它们。理论上这保持了相同的输出分布，但实践中接受率因领域和上下文而异。开箱即用的草稿模型在某些情况下可能正常工作，但在领域特定任务或非常长的上下文中通常会出现问题。

### 上下文窗口污染

在长时间的编码会话中，早期的错误会在上下文中积累。模型会看到自己的错误并可能延续它们。这是"Claude 变笨了"在单个会话内最常见的原因——不是模型退化，而是上下文污染。

**实用建议**：当感觉质量不对劲时，使用 `/compact` 或开启全新的会话。这是你可以做的最有效的一件事。

## 斯坦福研究——及其复杂性

2023 年斯坦福大学和加州大学伯克利分校的标志性研究（Chen、Zaharia、Zou）——"ChatGPT 的行为如何随时间变化？"——经常被引为 LLM 退化的证据。其头条发现是：

> GPT-4 在"这个数字是质数吗？逐步思考"任务上的准确率从 2023 年 3 月到 6 月从 **97.6% 下降到了 2.4%**。

### 该研究证明了什么

- 同一 LLM 服务的行为**可能在短时间内发生显著变化**
- 不同能力可能朝相反方向变化（GPT-4 数学能力下降，GPT-3.5 数学能力提升）
- 代码生成质量下降（GPT-4 可执行代码：52% → 10%）
- 该研究创造了术语 **"LLM 漂移"**

### 方法论批评

- 3 月版本使用了 **temperature 0.0**，而 6 月版本使用了 **temperature 1.0**——这是一个基本的混淆变量，增加了随机性
- 每个任务仅 **500 次查询**——对于确定性的统计结论来说样本量太小
- "数学问题"实际上是是/否题，模型的猜测模式发生了变化，而非数学能力本身
- 这些变化可能反映了有意的**训练后安全更新**，而非退化

该研究证明了重要的一点——LLM 行为随时间变化——但机制很可能是有意的更新，而非无意的退化。

## 心理因素

### 确认偏差

一旦有人在推特上说"今天的 Claude 很笨"，你就会开始注意每一个错误。在没有人抱怨的日子里，你会对同样的错误不以为意。社交媒体放大了这种效应。

### 蜜月效应

用户对新模型会经历一个初始的蜜月期，然后逐渐发现局限性。模型没有变化——期望值的上升速度超过了能力提升的速度。

### 任务难度差异

你的任务每天都不同。一天处理难题让你感觉模型变差了，即使实际情况并非如此。

### "周末 Claude"的迷思

尽管许多用户相信存在星期几的质量模式，但严格分析发现**没有一致证据**支持星期几的质量模式。一篇题为"AI 在星期一更笨"的分析结果是空手而归。

### LLM 的随机性本质

LLM 是概率性的。相同的提示词每次可能产生不同的输出。在运气不好的时候，你可能连续收到几个质量较差的响应——纯粹是随机性，而非退化。

---

## 结论

用户描述的现象是**真实存在的，但归因有误**：

- **正确**：他们的体验在某些日子里确实下降了
- **错误**：模型被有意"削弱"了

实际原因包括以下几种的组合：

1. **基础设施 Bug**——Anthropic 的事后分析已证实（影响高达 16% 的请求）
2. **MoE 路由差异**——Scale AI 测量的 ±8-14% 质量波动，即使零变更
3. **系统提示和训练后更新**——各提供商均有记录
4. **硬件异构性**——TPU vs GPU vs Trainium，各有不同的故障模式
5. **上下文污染**——长会话降低会话内质量
6. **确认偏差**——社交媒体放大感知到的模式
7. **随机性差异**——相同模型、相同提示、每次输出不同

测量问题非常严重：±8-14% 的日常差异意味着你无法区分真实的 5% 质量变化与噪声。这就是为什么"这都是你想多了"和"他们削弱了模型"两个阵营都感觉自信的原因——信噪比使得仅凭个人经验无法判断。

# 🔥 Codex 5.3 高推理与发现

### 报告范围

本节解释为什么用户可能在短时间内体验到 Claude 输出质量下降，而 Codex 5.3 在编码任务上感觉稳定或更强。重点不是永久性的模型质量排名，而是在真实服务条件下短期内的生产行为。

报告日期：2026 年 3 月 5 日。

### 观察到的模式

报告的模式是：

1. 模型质量在一段时间内可接受。
2. 质量似乎在几天内下降。
3. 质量恢复到接近之前的基线。

这种形态通常是服务栈或发布模式，而非永久性的基础模型能力变化。永久性的能力下降通常不会在没有显式回滚或修复的情况下如此快速地恢复。

### 高推理：为什么在糟糕窗口中 Codex 5.3 看起来更好

在另一家提供商的退化期内，Codex 5.3 可能因为几个同时发生的技术原因而显得明显更强：

1. **产品目标匹配**。Codex 5.3 针对代码生成和代理式编码工作流进行了优化，因此即使原始模型能力相当，由于工具编排、仓库推理和以代码为中心的指令微调，也能产生更好的编码效果。
2. **推理策略差异**。提供商独立调整延迟、推理深度和默认解码参数。同一天，一家提供商更保守的策略可能看起来比另一家激进的、速度优化的策略更"聪明"。
3. **服务路径分离**。即使两家提供商都托管了最先进的模型，它们运行不同的路由层、编译器/运行时栈和发布管道。一个栈中的事件并不意味着另一个栈中有关联的退化。
4. **发布和回滚时机**。如果一家提供商处于发布中期而另一家处于稳定状态，用户可能看到巨大的临时质量差异，而模型权重在底层没有任何长期变化。
5. **会话级污染效应**。在长时间的编码对话中，错误累积可能放大感知到的下降。一个竞争助手感觉更好仅仅是因为失败的会话被重置了，或者它的工具循环恢复得更快。

### 详细发现

对于"Claude 大约有四天感觉很弱，然后恢复了"这样的报告，最可能的解释是：

1. 提供商方的事件、路由问题、解码/运行时 Bug 或发布回归影响了部分请求。
2. 问题持续足够长的时间，在真实工作流中被反复注意到。
3. 问题被修复或回滚。
4. 感知到的质量迅速恢复。

在同一时期，Codex 5.3 可能感觉明显更好，因为它没有共享相同的事故路径，而且编码任务优化放大了实际效果上的差距。

### 此模式的假设排名

| 假设 | 可能性 | 理由 |
|------------|------------|-----------|
| 提供商事件加回滚 | 高 | 最匹配持续数天下降后快速恢复的模式 |
| 服务配置变更（采样/延迟/推理预算） | 高 | 无需模型重训练即可导致突然行为变化的常见来源 |
| 静默别名或快照迁移 | 中高 | 用户无可见操作即可改变行为 |
| 仅有提示漂移和上下文污染 | 中 | 可能降低会话质量，但不太可能单独解释广泛的多天报告 |
| 永久性基础模型退化 | 低 | 与快速恢复之前质量不一致 |

### 什么会证实或否定这一发现

要将这一发现从高置信度推理转变为硬证据，需要收集跨天同一任务集的请求级遥测数据：

1. 请求时的确切模型标识符和快照/别名。
2. 提供商暴露的任何后端指纹或发布标记。
3. 解码参数（temperature、top_p、top_k、最大 Token 数）。
4. 延迟、超时和错误率追踪。
5. 在固定编码基准提示集上的结构化质量评分。
6. 故障点的会话长度和 Token 上下文深度。

如果质量下降与事件窗口、配置变更或后端指纹变化相关，则事件/配置假设被确认。如果不存在此类变化且退化仅发生在长会话中，则上下文污染成为主要解释。

### 实用工程指导

为减少生产中的日常差异：

1. 尽可能固定模型快照，而不是使用浮动别名。
2. 存储请求元数据（模型 ID、参数、延迟、错误、响应质量标签）。
3. 每天运行固定的编码任务金丝雀测试集，并在回归时发出警报。
4. 在多次失败后重置或压缩长时间运行的会话。
5. 为事件窗口保留一个备用提供商/模型路径。
6. 在内部仪表板中将"模型质量"与"服务可靠性"分开。

### 最终结论

在 Claude 短期退化期间 Codex 5.3 表现更好，这是现代 LLM 运营中技术上合理且预期的结果。最强的解释不是永久性的模型崩溃，而是一家提供商临时的服务路径退化，结合了另一家提供商在同一时期的编码专属优化和稳定运营。
