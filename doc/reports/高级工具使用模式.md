## 概述

| 功能 | 解决的问题 | 令牌节省 | 可用性 |
|---------|---------------|---------------|--------------|
| 程序化工具调用 | 多步骤代理循环在往返中消耗令牌 | ~37% 减少 | API, Foundry (正式发布) |
| 动态过滤 | 网络搜索/获取的结果用无关内容膨胀上下文 | ~24% 更少输入令牌 | API, Foundry (正式发布) |
| 工具搜索工具 | 过多工具定义膨胀上下文 | ~85% 减少 | API, Foundry (正式发布) |
| 工具使用示例 | 仅凭 Schema 无法表达使用模式 | 72% → 90% 准确率 | API, Foundry (正式发布) |

所有功能自 **2026 年 2 月 18 日**起已**正式发布**。

**策略性分层** — 从最大的瓶颈开始：
- 工具定义导致上下文膨胀 → 工具搜索工具
- 大量中间结果 → 程序化工具调用
- 网络搜索噪音 → 动态过滤
- 参数错误 → 工具使用示例

## 程序化工具调用 (PTC)

<img src="assets/programmatic-tool-calling-diagram.svg" alt="PTC 示意图 — 传统 vs 程序化工具调用" width="100%" />

### 范式转变

**之前（传统工具调用）：**
```
用户提示 → Claude → 工具调用 1 → 响应 1 → Claude → 工具调用 2 → 响应 2 → Claude → 工具调用 3 → 响应 3 → Claude → 最终答案
```
每次工具调用需要一次完整的模型往返。3 个工具 = 3 次推理通过。

**之后（程序化工具调用）：**
```
用户提示 → Claude → 编写 Python 脚本 → 脚本内部调用工具 1、工具 2、工具 3 → stdout → Claude → 最终答案
```
Claude 编写代码来编排所有工具。只有最终的 `stdout` 进入上下文窗口。3 个工具 = 1 次推理通过。

### 工作原理

1. 您定义工具时添加 `allowed_callers: ["code_execution_20250825"]`
2. Claude 编写 Python 代码，在沙箱中将这些工具作为异步函数调用
3. 当调用工具函数时，沙箱暂停，API 返回一个 `tool_use` 块
4. 您提供工具结果 — 结果发送给**正在运行的代码**，而非 Claude 的上下文
5. 代码恢复执行，处理结果，如果需要则调用更多工具
6. 只有最终执行的 `stdout` 到达 Claude

### 关键配置

```json
{
  "tools": [
    {
      "type": "code_execution_20250825",
      "name": "code_execution"
    },
    {
      "name": "query_database",
      "description": "执行 SQL 查询。以 JSON 对象形式返回行，字段包括：id (str)、name (str)、revenue (float)。",
      "input_schema": {
        "type": "object",
        "properties": {
          "sql": { "type": "string", "description": "要执行的 SQL 查询" }
        },
        "required": ["sql"]
      },
      "allowed_callers": ["code_execution_20250825"]
    }
  ]
}
```

### `allowed_callers` 字段

| 值 | 行为 |
|-------|----------|
| `["direct"]` | 仅传统工具调用（省略时的默认值） |
| `["code_execution_20250825"]` | 仅可从 Python 沙箱调用 |
| `["direct", "code_execution_20250825"]` | 两种模式均可用 |

**建议：** 每个工具选择一种模式，不要同时选择两种。这为 Claude 提供更清晰的指导。

### 响应中的 `caller` 字段

每个工具使用块包含一个 `caller` 字段，让您知道它是如何被调用的：

```json
// 直接（传统）
{ "caller": { "type": "direct" } }

// 程序化（来自代码执行）
{ "caller": { "type": "code_execution_20250825", "tool_id": "srvtoolu_abc123" } }
```

### 高级模式

**批量处理** — 在 1 次推理通过中处理 N 个项目：
```python
regions = ["West", "East", "Central", "North", "South"]
results = {}
for region in regions:
    data = await query_database(f"SELECT SUM(revenue) FROM sales WHERE region='{region}'")
    results[region] = data[0]["revenue"]

top = max(results.items(), key=lambda x: x[1])
print(f"Top region: {top[0]} with ${top[1]:,}")
```

**提前终止** — 一旦满足成功条件就停止：
```python
endpoints = ["us-east", "eu-west", "apac"]
for endpoint in endpoints:
    status = await check_health(endpoint)
    if status == "healthy":
        print(f"Found healthy endpoint: {endpoint}")
        break
```

**条件性工具选择：**
```python
file_info = await get_file_info(path)
if file_info["size"] < 10000:
    content = await read_full_file(path)
else:
    content = await read_file_summary(path)
print(content)
```

**数据过滤** — 减少 Claude 看到的内容：
```python
logs = await fetch_logs(server_id)
errors = [log for log in logs if "ERROR" in log]
print(f"Found {len(errors)} errors")
for error in errors[-10:]:
    print(error)
```

### 模型兼容性

| 模型 | 支持 |
|-------|-----------|
| Claude Opus 4.6 | 是 |
| Claude Sonnet 4.6 | 是 |
| Claude Sonnet 4.5 | 是 |
| Claude Opus 4.5 | 是 |

### 约束条件

| 约束 | 详情 |
|-----------|--------|
| **不在 Bedrock/Vertex 上** | 仅限 API 和 Foundry |
| **无 MCP 工具** | MCP 连接器工具不能以编程方式调用 |
| **无网络搜索/获取** | PTC 中不支持网络工具 |
| **无结构化输出** | 不兼容 `strict: true` 工具 |
| **无强制工具选择** | `tool_choice` 不能强制 PTC |
| **容器生命周期** | 过期前约 4.5 分钟 |
| **ZDR** | 不在零数据保留覆盖范围内 |
| **工具结果为字符串** | 验证外部结果以防止代码注入风险 |

### 何时使用 PTC

| 好的使用场景 | 不太理想的场景 |
|----------------|------------|
| 处理需要聚合结果的大型数据集 | 带有简单响应的单次工具调用 |
| 连续 3 次以上依赖的工具调用 | 需要立即用户反馈的工具 |
| 在 Claude 看到之前过滤/转换结果 | 非常快速的操作（开销 > 收益） |
| 跨多个项目的并行操作 | |
| 基于中间结果的条件逻辑 | |

### 令牌效率

- 程序化调用的工具结果**不会添加到 Claude 的上下文中** — 仅最终 `stdout`
- 中间处理在代码中进行，而非模型令牌
- 10 个程序化工具 ≈ 10 次直接调用的 1/10 令牌量

## 网络搜索/获取的动态过滤

### 问题

网络搜索和获取工具将完整的 HTML 页面转储到 Claude 的上下文窗口中。其中大部分内容是不相关的 — 导航、广告、样板代码。Claude 随后对所有内容进行推理，浪费令牌并降低准确性。

### 解决方案

Claude 现在**编写并执行 Python 代码来过滤网络结果**，然后再将其送入上下文窗口。Claude 不再对原始 HTML 进行推理，而是在沙箱中过滤、解析和提取仅相关的内容。

### 工作原理

**之前：**
```
查询 → 搜索结果 → 获取完整 HTML × N 页 → 所有内容进入上下文 → Claude 对所有内容进行推理
```

**之后：**
```
查询 → 搜索结果 → Claude 编写过滤代码 → 代码仅提取相关内容 → 过滤后的结果进入上下文
```

### API 配置

使用带有 beta 标头的更新版工具类型：

```json
{
  "model": "claude-opus-4-6",
  "max_tokens": 4096,
  "tools": [
    {
      "type": "web_search_20260209",
      "name": "web_search"
    },
    {
      "type": "web_fetch_20260209",
      "name": "web_fetch"
    }
  ]
}
```

**所需标头：** `anthropic-beta: code-execution-web-tools-2026-02-09`

**默认启用** — 在使用新版本工具类型时，搭配 Sonnet 4.6 和 Opus 4.6 时默认开启。

### 基准测试结果

**BrowseComp**（在网站上查找特定信息）：

| 模型 | 无过滤 | 有过滤 | 改进 |
|-------|-------------------|----------------|-------------|
| Sonnet 4.6 | 33.3% | **46.6%** | +13.3 个百分点 |
| Opus 4.6 | 45.3% | **61.6%** | +16.3 个百分点 |

**DeepsearchQA**（多步骤研究，F1 分数）：

| 模型 | 无过滤 | 有过滤 | 改进 |
|-------|-------------------|----------------|-------------|
| Sonnet 4.6 | 52.6% | **59.4%** | +6.8 个百分点 |
| Opus 4.6 | 69.8% | **77.3%** | +7.5 个百分点 |

**令牌效率：** 平均减少 24% 的输入令牌。Sonnet 4.6 可降低成本；Opus 4.6 可能略有增加，因为需要更复杂的过滤代码。

### 使用场景

- 筛选技术文档
- 跨多个来源验证引用
- 交叉引用搜索结果
- 多步骤研究查询
- 在大型页面中查找隐藏的特定数据点

## 工具搜索工具

### 问题

预先加载所有工具定义浪费上下文。如果您有 50 个 MCP 工具，每个约 1.5K 令牌，那就是在用户提问之前就占用了 75K 令牌。

### 解决方案

将不常用的工具标记为 `defer_loading: true`。它们被排除在初始上下文之外。Claude 通过工具搜索工具按需发现它们。

### 配置

```json
{
  "tools": [
    {
      "type": "mcp_toolset",
      "mcp_server_name": "google-drive",
      "default_config": { "defer_loading": true },
      "configs": {
        "search_files": { "defer_loading": false }
      }
    }
  ]
}
```

### 最佳实践

- 始终加载 3-5 个最常用的工具，其余工具延后加载
- 编写清晰、描述性的工具名称和描述（搜索依赖于它们）
- 在系统提示中记录可用的能力

### 何时使用

- 工具定义消耗超过 10K 令牌
- 有 10 个以上工具可用
- 有多个 MCP 服务器
- 因选项过多导致工具选择准确性问题

### 令牌节省

工具定义令牌减少约 85%（在 Anthropic 的基准测试中从 77K 降至 8.7K）。

### Claude Code 中的等价功能

Claude Code 具有 **MCP 工具搜索自动模式**（自 v2.1.7 起默认启用）。当 MCP 工具描述超过上下文的 10% 时，它们会被延后并通过 `MCPSearch` 发现。通过 `ENABLE_TOOL_SEARCH=auto:N` 配置阈值，其中 N 是上下文百分比（0-100）。

## 工具使用示例

### 问题

JSON Schema 定义了结构，但无法表达：
- 何时包含可选参数
- 哪些参数组合有意义
- 格式约定（日期格式、ID 模式）
- 嵌套结构的使用方式

### 解决方案

在工具定义中添加 `input_examples` — 超越 schema 的具体使用模式。

### 配置

```json
{
  "name": "create_ticket",
  "description": "创建支持工单",
  "input_schema": {
    "type": "object",
    "properties": {
      "title": { "type": "string" },
      "priority": { "type": "string", "enum": ["low", "medium", "high", "critical"] },
      "assignee": { "type": "string" },
      "labels": { "type": "array", "items": { "type": "string" } }
    },
    "required": ["title"]
  },
  "input_examples": [
    {
      "title": "登录页面返回 500 错误",
      "priority": "critical",
      "assignee": "oncall-team",
      "labels": ["bug", "auth", "production"]
    },
    {
      "title": "添加暗黑模式支持",
      "priority": "low",
      "labels": ["feature-request", "ui"]
    },
    {
      "title": "更新 v2 端点的 API 文档"
    }
  ]
}
```

### 最佳实践

- 使用**真实数据**，而非像 "example_value" 这样的占位字符串
- 展示**多样性**：最小、部分和完整规范
- 保持简洁：**每个工具 1-5 个示例**
- 专注于消除歧义 — 以行为清晰度为目标，而非 schema 的完备性
- 展示参数关联性（例如，`priority: "critical"` 往往有 `assignee`）

### 效果

在 Anthropic 的基准测试中，复杂参数处理准确率从 72% 提升至 90%。

## 与 Claude Code 的相关性

### 直接适用于 Claude Code 用户的功能

| 功能 | Claude Code 状态 | 操作建议 |
|---------|-------------------|--------|
| 工具搜索 | 自 v2.1.7 起内置为 MCPSearch 自动模式 | 如果有很多 MCP 工具，调整 `ENABLE_TOOL_SEARCH=auto:N` |
| 动态过滤 | CLI 中不可用（API 级别的网络工具） | 对于使用 Agent SDK 进行网络研究的人员相关 |
| PTC | CLI 中不可用 | 对于使用 Agent SDK 构建自定义代理的人员相关 |
| 工具使用示例 | CLI 中不可配置 | 对于自定义 MCP 服务器的作者相关 |

### 对于 Agent SDK 开发者

如果您使用 `@anthropic-ai/claude-agent-sdk` 构建代理，PTC 可以立即投入使用：

1. 将 `code_execution_20250825` 添加到工具数组中
2. 在受益于批处理/过滤的工具上设置 `allowed_callers`
3. 实现工具结果循环（暂停 → 提供结果 → 恢复）
4. 从工具返回结构化数据（JSON），以便更容易地以编程方式解析

### 对于 MCP 服务器作者

如果您正在构建自定义 MCP 服务器，工具使用示例可以改善 Claude 使用您工具的方式：
- 在工具 schema 中添加 `input_examples`
- 在描述中清晰记录返回格式（PTC 需要解析它们）
